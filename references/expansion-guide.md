# KKSK Expansion Guide

Read this after Kime succeeds and the function is adopted. This guide covers the decision of what to build next.

## The Core Question

When the function works, ask: **does the next pain touch this function's output?**

- If yes → run KKSK again. The output of the first function becomes the input context for the next cycle.
- If no → you're done, or you need a different framework entirely.

This creates natural composition: small, proven functions chained together, each validated by actual use before the next one is built.

## Deepen vs. Go Adjacent

**Deepen** if your current users have a *second* pain that touches the output of your first function. This is the natural KKSK chain: the output of cycle 1 becomes the input context for cycle 2, serving the same wedge audience.

**Go adjacent** if your wedge audience is saturated and a *neighboring* segment has started pulling your solution toward them unprompted. The key word is "pulling" — if you have to push, it's too early.

### How to Tell the Difference

| Signal | Meaning |
|---|---|
| Users ask "can it also do X?" where X relates to the output | Deepen — run KKSK again for the same audience |
| Users from a different context start using the tool without modification | Go adjacent — the wedge is proving transferable |
| Users ask "can it also do X?" where X is unrelated to the output | Neither — this is scope creep, not expansion signal |
| You *think* another market would benefit but nobody's asked | Too early — keep serving the wedge |

## The Trojan Horse Test

Can your success in the wedge market serve as proof that opens the next market without you changing the product?

- If yes → expand. Your wedge success is your sales pitch.
- If no → if expanding requires rebuilding for a different buyer, run KKSK from scratch. You're starting a new wedge, not extending the old one.

## Anti-Patterns

**Simultaneous expansion:** "We've won Higher Ed, let's go after Enterprise and Consumer simultaneously." This is indigestion. Pick one adjacent market. Win it. Then reassess.

**Push-based expansion:** If you have to convince the next market that they need your tool, the pull isn't there yet. Go back and deepen with your existing audience.

**Feature-driven expansion:** Adding features to attract a new market instead of finding a new market that your existing function already serves. This reverses the KKSK logic — the function should follow the pain, not the other way around.

## The Expansion KKSK Cycle

When expanding, run the full KKSK cycle for the new context:

1. **Kamae:** What is the new audience's pain? (Not your existing audience's pain applied to new people — their *actual* pain in *their* workflow.)
2. **Kuzushi:** How does the new audience currently survive? (Their workarounds will be different.)
3. **Sasoi:** Is it the same function, or does it need modification? (If same → true expansion. If different → new wedge.)
4. **Kime:** Build only what's different. Reuse what's proven.
